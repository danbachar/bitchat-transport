\section{Introduction}
% =============================================================================

SGN uses two transport layers---\textbf{BLE} (short-range, no Internet) and \textbf{libp2p} (UDX/TCP over UDP, global reach). All messages are \textbf{direct}---delivered over a connection between sender and recipient. Routable friends (peers with globally routable IP addresses) provide \textbf{signaling} to help NATed peers establish direct connections via UDP hole-punching.

\textbf{Key constraints}: (1)~\textbf{Fair message delivery}: every \texttt{send()} is eventually delivered---the sender queues locally when the recipient is temporarily unreachable; (2)~Messages are always direct (no forwarding through intermediate peers); (3)~Both BLE and libp2p are always enabled.

\textbf{Address types}: BLE-only (no IP), Private IPv4 (PIv4, behind NAT), Public IPv4 (PuIv4), Link-local IPv6 (LLv6, not routable), Public IPv6 (PuIv6).

\textbf{NAT types} (relevant for PIv4 peers behind NAT):
\begin{itemize}[nosep]
  \item \textbf{Full-cone NAT}: Once an internal host has sent a packet out, \emph{any} external host can send packets back to the mapped address:port. Easiest to traverse.
  \item \textbf{Restricted-cone NAT}: Like full-cone, but only allows incoming packets from an external IP that the internal host has previously sent to. The port does not matter.
  \item \textbf{Port-restricted cone NAT}: Like restricted-cone, but also checks the external port. The internal host must have sent to that exact IP:port pair.
  \item \textbf{Symmetric NAT}: Creates a \emph{different} external mapping for each destination IP:port pair. Because the mapped port is unpredictable, a third party (routable friend) cannot tell peer B which port to send to---hole-punching fails. Symmetric$\leftrightarrow$symmetric is the worst case; symmetric$\leftrightarrow$cone sometimes works with port prediction.
\end{itemize}
Hole-punching (Section~\ref{sec:routable-friend}) works reliably for all three cone types. Symmetric NAT$\leftrightarrow$symmetric NAT cannot be hole-punched; sender queues locally until a direct path becomes available.

\textbf{Topologies}: T1~(co-located, $<$30m), T2~(remote), T3~(routable friend --- signaling).

% =============================================================================
\section{Transport Mechanics}
% =============================================================================

\textbf{BLE}: 10--100m range, no IP required. Each device advertises a GATT service UUID with a fixed 8-byte Grassroots prefix (enables scan filtering) and an 8-byte agent suffix derived from the Ed25519 public key (SHA-256 hash). After connection, ANNOUNCE packets are exchanged (full 32-byte public key, nickname, IP addresses).

\textbf{LibP2P}: UDX (primary, UDP-based) and TCP (fallback) transports, secured with \textbf{Noise XX} (Curve25519 DH, ChaCha20-Poly1305, SHA-256) providing mutual authentication and forward secrecy. LibP2P generates a \texttt{PeerId} from the Ed25519 key; the SGN public key remains the canonical identity. DHT disabled; mDNS enabled for LAN discovery. NAT traversal via routable-friend-coordinated UDP hole-punching (see Section~\ref{sec:routable-friend}). When no direct path exists, the sender queues locally until one becomes available.

\subsection{T3: Routable Friend (Signaling)}
\label{sec:routable-friend}

A \textbf{routable friend} is an ordinary SGN agent that currently has a \textbf{globally routable IP address}. It is not a server---it is a regular friend running the same app, who happens to be directly reachable from the Internet right now.

\textbf{Who can be a routable friend?}
\begin{itemize}[nosep]
  \item \textbf{Smartphones on carriers with public IPv6} (increasingly common; many LTE/5G carriers assign PuIv6)
  \item \textbf{Devices on networks with public IPv4} (less common for mobile, common for fiber/business)
  \item \textbf{Dedicated always-on servers} (for reliability, but not required)
\end{itemize}

\textbf{Signaling responsibilities}:
\begin{enumerate}[nosep]
  \item \textbf{Address registration}: NATed friends connect and report their current public IP:port.
  \item \textbf{Address notification}: When A wants to reach B, the routable friend provides A with B's registered address (and vice versa).
  \item \textbf{Hole-punch coordination}: Instructs both agents to begin simultaneous UDP sends, opening NAT mappings.
\end{enumerate}

\textbf{Symmetric NAT}: When hole-punching fails (both peers behind symmetric NAT), the peers cannot establish a direct connection remotely. The sender queues locally; delivery occurs when a direct path becomes available (e.g., co-location via BLE, or network conditions change).

\textbf{No SPOF}: The model is inherently redundant. An agent befriends multiple routable friends; if one goes offline, others provide signaling. Any new friend with a public IP automatically becomes a potential routable friend.

\subsubsection{Discovering Routable Friends}

How does agent A know which friends currently have a routable address?

\begin{enumerate}[nosep]
  \item \textbf{ANNOUNCE packets}: Each peer includes its current IP addresses in ANNOUNCE. Friends see whether the address is globally routable and mark that peer as a potential routable friend.
  \item \textbf{Address change notifications}: When a peer's network changes (WiFi$\to$LTE, gaining/losing PuIv6), it pushes an updated address set to all connected friends.
  \item \textbf{Probing}: If A has a stale address for friend C, A attempts a UDX connection. If it succeeds, C is reachable. If not, A tries the next known routable friend.
\end{enumerate}

The set of routable friends is inherently dynamic---friends gain and lose routable addresses as they move between networks. Redundancy absorbs this churn.

\subsubsection{Mutual Routable Friend Requirement}

For hole-punch coordination, A and B need a \emph{mutual} routable friend S simultaneously connected to both:

\begin{itemize}[nosep]
  \item S must know both A's and B's current public IP:port (both must have registered with S).
  \item If A and B share no mutual routable friend, hole-punching cannot be coordinated. Sender queues locally until a direct path becomes available.
  \item In practice, the social graph grows through BLE encounters. Two remote friends likely share mutual friends. The introduction protocol (GLP application layer) can explicitly introduce friends to routable peers.
\end{itemize}

\subsubsection{IPv6 Firewalling}

Some carriers assign globally routable IPv6 addresses but \textbf{firewall incoming UDP}. The phone has a PuIv6 address but cannot receive unsolicited packets---a false positive.

\textbf{Detection}: When a peer detects it has a PuIv6 address, it asks a connected friend to attempt an incoming UDP connection. If it succeeds, the peer is genuinely routable and advertises itself as such. If it fails (firewalled), it does not. This is analogous to libp2p's AutoNAT ``dial me back'' probe.

\textbf{No connection migration is libp2p's biggest weakness for mobile.} When IP changes, all UDX/TCP connections die and must reconnect from scratch (re-register with routable friends, re-coordinate hole-punches). The sender's local queue preserves messages during the gap, ensuring fair delivery once connectivity is restored.

BLE and libp2p are \textbf{complementary}: BLE provides resilient local connectivity immune to network configuration; libp2p provides global reach immune to physical proximity. \textbf{Local queuing at the sender} ensures fair message delivery across transient disconnections---messages are always delivered directly once a path is available.

\textbf{Primary gaps}:
\begin{enumerate}[nosep]
  \item \textbf{No connection migration}---libp2p UDX/TCP connections die on IP change. QUIC-based transport would solve this.
  \item \textbf{Cross-protocol bridging}---IPv4$\leftrightarrow$IPv6 needs a dual-stack routable friend for hole-punch coordination.
  \item \textbf{Symmetric NAT}---both peers behind symmetric NAT cannot hole-punch; must wait for a direct path (BLE co-location or network change).
  \item \textbf{PeerId $\neq$ SGN public key}---libp2p derives PeerId from the Ed25519 key, but maintaining the bidirectional mapping adds complexity.
\end{enumerate}

\textbf{Resolved gaps} (vs.\ previous analysis):
\begin{itemize}[nosep]
  \item \textbf{Bootstrap SPOF}: Eliminated. Routable friends are federated---any friend with a public IP. No single point of failure.
  \item \textbf{No store-and-forward}: Resolved. Sender-side local queuing provides fair message delivery.
\end{itemize}

\textbf{Recommendations}: (1)~Evaluate QUIC-based transport for connection migration; (2)~Unify PeerId with SGN public key if possible; (3)~Enable address exchange via QR code/link for remote-first friendships.

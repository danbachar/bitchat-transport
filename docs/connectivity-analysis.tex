\section{Introduction}
% =============================================================================

SGN uses two transport layers---\textbf{BLE} (short-range, no Internet) and \textbf{libp2p} (UDX/TCP over UDP, global reach)---plus \textbf{application-level peer relay} where a connected peer C forwards end-to-end encrypted messages between A and B when no direct path exists.

\textbf{Key constraints}: (1)~\textbf{Fair message delivery}: every \texttt{send()} is eventually delivered---the sender queues locally when the recipient is temporarily unreachable; (2)~Single-hop relay only (A$\to$C$\to$B, no multi-hop); relay peers do NOT queue---forwarding is real-time only; (3)~Transport priority: BLE $\to$ libp2p (direct) $\to$ peer relay.

\textbf{Address types}: BLE-only (no IP), Private IPv4 (PIv4, behind NAT), Public IPv4 (PuIv4), Link-local IPv6 (LLv6, not routable), Public IPv6 (PuIv6).

\textbf{Topologies}: T1~(co-located, $<$30m), T2~(remote), T3~(friend in the middle --- relay and signaling).

% =============================================================================
\section{Transport Mechanics}
% =============================================================================

\textbf{BLE}: 10--100m range, no IP required. Each device advertises a GATT service UUID with a fixed 8-byte Grassroots prefix (enables scan filtering) and an 8-byte agent suffix derived from the Ed25519 public key (SHA-256 hash). After connection, ANNOUNCE packets are exchanged (full 32-byte public key, nickname, IP addresses).

\textbf{LibP2P}: UDX (primary, UDP-based) and TCP (fallback) transports, secured with \textbf{Noise XX} (Curve25519 DH, ChaCha20-Poly1305, SHA-256) providing mutual authentication and forward secrecy. LibP2P generates a \texttt{PeerId} from the Ed25519 key; the SGN public key remains the canonical identity. DHT disabled; mDNS enabled for LAN discovery. NAT traversal via routable-friend-coordinated UDP hole-punching (see Section~\ref{sec:routable-friend}).

\textbf{Application-level peer relay}: Any connected peer C can forward messages across transport boundaries (e.g., BLE$\to$libp2p). C sees only encrypted envelopes. Single hop, real-time only (C does not queue). The \emph{sender} queues locally if no path exists, preserving fair message delivery.

\textbf{Peer relay impact}: Significant for BLE-only peers. If a BLE-only Peer A has a co-located Peer C with libp2p, C can forward to the remote Peer B---transforming ``unreachable'' into reachable.

\textbf{NAT traversal}: Hole-punching is coordinated by a mutual routable friend (Section~\ref{sec:routable-friend}). Works for cone NAT (full, restricted, port-restricted). Symmetric NAT$\leftrightarrow$symmetric NAT fails hole-punching; falls back to app-level peer relay through the routable friend's social graph.

\subsection{T3: Friend in the Middle (Relay + Signaling)}
\label{sec:routable-friend}

A third peer C exists between peers A and B. C can help in two ways, depending on whether C has a \textbf{globally routable IP address}:

\begin{enumerate}[nosep]
  \item \textbf{Message relay}: C forwards end-to-end encrypted messages from A to B (or vice versa). Any peer can do this regardless of IP type. C sees only encrypted envelopes.
  \item \textbf{Signaling}: C coordinates UDP hole-punching between A and B so they can establish a direct connection. This requires C to have a globally routable address (PuIv6 or PuIv4)---making C a \textbf{routable friend}.
\end{enumerate}

A friend with a routable address can do \emph{both} simultaneously. Signaling establishes direct connections (reducing the need for relay), while relay provides a fallback when signaling fails (e.g., symmetric NAT).

\subsubsection{Signaling (Routable Friends)}

A \textbf{routable friend} is an ordinary SGN agent that currently has a \textbf{globally routable IP address}. It is not a server---it is a regular friend running the same app, who happens to be directly reachable from the Internet right now.

\textbf{Who can be a routable friend?}
\begin{itemize}[nosep]
  \item \textbf{Smartphones on carriers with public IPv6} (increasingly common; many LTE/5G carriers assign PuIv6)
  \item \textbf{Devices on networks with public IPv4} (less common for mobile, common for fiber/business)
  \item \textbf{Dedicated always-on servers} (for reliability, but not required)
\end{itemize}

\textbf{Signaling responsibilities}:
\begin{enumerate}[nosep]
  \item \textbf{Address registration}: NATed friends connect and report their current public IP:port.
  \item \textbf{Address notification}: When A wants to reach B, the routable friend provides A with B's registered address (and vice versa).
  \item \textbf{Hole-punch coordination}: Instructs both agents to begin simultaneous UDP sends, opening NAT mappings.
\end{enumerate}

\textbf{Symmetric NAT fallback}: When hole-punching fails (both peers behind symmetric NAT), the routable friend falls back to \emph{message relay}---it is just another friend in the social graph.

\textbf{No SPOF}: The model is inherently redundant. An agent befriends multiple routable friends; if one goes offline, others provide signaling. Any new friend with a public IP automatically becomes a potential routable friend.

\subsubsection{Discovering Routable Friends}

How does agent A know which friends currently have a routable address?

\begin{enumerate}[nosep]
  \item \textbf{ANNOUNCE packets}: Each peer includes its current IP addresses in ANNOUNCE. Friends see whether the address is globally routable and mark that peer as a potential routable friend.
  \item \textbf{Address change notifications}: When a peer's network changes (WiFi$\to$LTE, gaining/losing PuIv6), it pushes an updated address set to all connected friends.
  \item \textbf{Probing}: If A has a stale address for friend C, A attempts a UDX connection. If it succeeds, C is reachable. If not, A tries the next known routable friend.
\end{enumerate}

The set of routable friends is inherently dynamic---friends gain and lose routable addresses as they move between networks. Redundancy absorbs this churn.

\subsubsection{Mutual Routable Friend Requirement}

For hole-punch coordination, A and B need a \emph{mutual} routable friend S simultaneously connected to both:

\begin{itemize}[nosep]
  \item S must know both A's and B's current public IP:port (both must have registered with S).
  \item If A and B share no mutual routable friend, hole-punching cannot be coordinated. Communication falls back to message relay through whatever peers connect A to B.
  \item In practice, the social graph grows through BLE encounters. Two remote friends likely share mutual friends. The introduction protocol (GLP application layer) can explicitly introduce friends to routable peers.
\end{itemize}

\subsubsection{IPv6 Firewalling}

Some carriers assign globally routable IPv6 addresses but \textbf{firewall incoming UDP}. The phone has a PuIv6 address but cannot receive unsolicited packets---a false positive.

\textbf{Detection}: When a peer detects it has a PuIv6 address, it asks a connected friend to attempt an incoming UDP connection. If it succeeds, the peer is genuinely routable and advertises itself as such. If it fails (firewalled), it does not. This is analogous to libp2p's AutoNAT ``dial me back'' probe.

\textbf{No connection migration is libp2p's biggest weakness for mobile.} When IP changes, all UDX/TCP connections die and must reconnect from scratch (re-register with routable friends, re-coordinate hole-punches). A co-located BLE peer can relay during the blackout. The sender's local queue preserves messages during the gap, ensuring fair delivery once connectivity is restored.

BLE, libp2p, and peer relay are \textbf{complementary}: BLE provides resilient local connectivity immune to network configuration; libp2p provides global reach immune to physical proximity; peer relay bridges the two. \textbf{Local queuing at the sender} ensures fair message delivery across transient disconnections.

\textbf{Primary gaps}:
\begin{enumerate}[nosep]
  \item \textbf{No connection migration}---libp2p UDX/TCP connections die on IP change. QUIC-based transport would solve this.
  \item \textbf{Cross-protocol bridging}---IPv4$\leftrightarrow$IPv6 needs a dual-stack routable friend; app relay partially mitigates.
  \item \textbf{Relay peer availability}---sparse networks may lack suitable relay peers; multiple routable friends mitigate.
  \item \textbf{Single relay hop limit}---deliberate simplicity trade-off; see future work.
  \item \textbf{PeerId $\neq$ SGN public key}---libp2p derives PeerId from the Ed25519 key, but maintaining the bidirectional mapping adds complexity.
\end{enumerate}

\textbf{Resolved gaps} (vs.\ previous analysis):
\begin{itemize}[nosep]
  \item \textbf{Bootstrap SPOF}: Eliminated. Routable friends are federated---any friend with a public IP. No single point of failure.
  \item \textbf{No store-and-forward}: Resolved. Sender-side local queuing provides fair message delivery. Relay peers still do not queue (real-time forwarding only).
\end{itemize}

\textbf{Recommendations}: (1)~Evaluate QUIC-based transport for connection migration; (2)~Unify PeerId with SGN public key if possible; (3)~Enable address exchange via QR code/link for remote-first friendships.

\section{Future Work: Geographic-Optimized Multi-Hop Routing}
% =============================================================================

The single-hop relay constraint (Section~3.3) exists because standard multi-hop routing (e.g., AODV) requires flooding-based route discovery---broadcasting RREQ to all peers. In a mobile BLE mesh with volatile topology, this is expensive and unreliable. \textbf{Geohash-guided AODV} can make multi-hop viable by constraining route discovery spatially.

\subsection{Geohash-Guided AODV}

\begin{enumerate}[nosep]
  \item \textbf{Geohash in ANNOUNCE}: Each peer includes a truncated geohash (5--6 characters, $\sim$5\,km$^2$ precision) in its ANNOUNCE packet. All peers maintain a \texttt{nodeId $\to$ geohash} mapping.

  \item \textbf{Directional RREQ}: When A needs a route to B and knows B's approximate geohash, A tags the RREQ with B's geohash. Intermediate peers only re-broadcast if their own geohash is geographically \emph{closer} to the destination (or within a configurable angular cone). This prunes the flood dramatically compared to standard AODV.

  \item \textbf{Geohash ring expansion}: If directional RREQ fails (no peers in the cone), reduce geohash precision (fewer characters = larger area) and retry---analogous to AODV's expanding ring search, but spatially informed.

  \item \textbf{Route maintenance}: Standard AODV RERR messages when a hop breaks. Geo-awareness helps select alternate next-hops that are still in the right geographic direction.
\end{enumerate}

\textbf{Why geohash?} Hierarchical (truncating characters widens the area---no distance math), prefix-matchable (shared prefix = spatial proximity), privacy-preserving (5-char $\approx$ 5\,km$^2$, reveals neighborhood, not location), and compact (5--6 ASCII characters per ANNOUNCE).

\subsection{Open Questions}

\begin{itemize}[nosep]
  \item \textbf{Location permission}: Mobile OS requires explicit consent; geohash must be opt-in.
  \item \textbf{Hybrid routing}: Geo-AODV only for BLE mesh (physical proximity); libp2p for long-range. Geographic routing is irrelevant for Internet transport.
  \item \textbf{Geohash spoofing}: A peer could lie about its geohash to attract or deflect traffic. Mitigations: cross-reference BLE RSSI, require signed geohash attestations from multiple neighbors.
  \item \textbf{TTL cap}: Practical limit likely 3--4 hops before latency and reliability degrade unacceptably.
\end{itemize}

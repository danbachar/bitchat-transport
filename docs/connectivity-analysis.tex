\section{Introduction}
% =============================================================================

This document provides a comprehensive analysis of connectivity between Social Grassroots Networking (SGN) peers across all combinations of network address types, peer topologies, and mobility scenarios. SGN uses two transport layers:

\begin{itemize}
  \item \textbf{BLE (Bluetooth Low Energy)} --- short-range, no Internet required, used for local peer discovery and communication
  \item \textbf{Internet transport} --- historically \textbf{libp2p} (UDX/TCP), now \textbf{iroh} (QUIC/UDP). Used for global connectivity beyond BLE range.
\end{itemize}

This document analyzes the libp2p-based transport as the primary architecture, then presents the migration to iroh and evaluates how each connectivity scenario changed.

\subsection{Key Constraints}

\begin{enumerate}
  \item \textbf{No store-and-forward}: If a peer is unreachable (even via relay), the message fails. No caching, no queuing. Relay is real-time only.
  \item \textbf{Single-hop peer relay}: A connected peer C \textbf{may} forward messages from A to B if C can reach B. Messages are end-to-end encrypted; C cannot read them. Maximum one relay hop (no multi-hop chains A$\to$C$\to$D$\to$B).
  \item \textbf{Transport priority}: BLE first, then iroh (direct), then peer relay (last resort).
\end{enumerate}

\subsection{Peer Topology Definitions}

We evaluate four peer topologies:

\begin{enumerate}[label=\textbf{T\arabic*}]
  \item \textbf{Friend Next to Me} --- Both peers are physically co-located ($<$30m). Both BLE and Internet transport are potentially available.
  \item \textbf{Friend Far Away} --- Peers are geographically separated. Only Internet transport is possible. No BLE.
  \item \textbf{Potential Relay Peer (``Friend in the Middle'')} --- A third peer C exists between peers A and B. We evaluate whether C can facilitate A$\leftrightarrow$B communication.
  \item \textbf{Server} --- Either (a) public relay/bootstrap infrastructure, or (b) a custom always-on supernode with a stable public IP.
\end{enumerate}

\section{Transport Mechanics}
% =============================================================================

\subsection{BLE Transport}

\begin{itemize}
  \item \textbf{Range}: Typically 10--30m indoors, up to 100m outdoors with line of sight.
  \item \textbf{Discovery}: Each device advertises a unique service UUID derived from its Ed25519 public key. Scanning discovers all nearby BLE devices; service discovery confirms SGN peers.
  \item \textbf{Data}: After GATT service discovery and ANNOUNCE exchange, data flows over the SGN GATT characteristic.
  \item \textbf{No IP required}: BLE operates below the IP layer. Works without any Internet or WiFi connectivity.
  \item \textbf{Symmetric}: Both peers must be in range; there is no multi-hop BLE.
  \item \textbf{Unchanged across stacks}: BLE behavior is identical under both libp2p and iroh. It is independent of the Internet transport.
\end{itemize}

\subsection{LibP2P Transport (Original Stack)}

\begin{itemize}
  \item \textbf{Transports}: Two transport protocols were available:
    \begin{itemize}
      \item \textbf{UDX} (UDP-based, custom protocol) --- primary transport, preferred for performance
      \item \textbf{TCP} --- secondary, used for IPFS ecosystem interoperability and as firewall fallback
    \end{itemize}
  \item \textbf{Security}: Noise protocol for all connections (both UDX and TCP).
  \item \textbf{Identity}: LibP2P used its own Ed25519 keypair to generate a \texttt{PeerId}, \textbf{separate} from the SGN application-layer Ed25519 public key. This required maintaining a double mapping:
    \begin{enumerate}
      \item BLE Device ID $\leftrightarrow$ LibP2P PeerId
      \item LibP2P PeerId $\leftrightarrow$ SGN Public Key
    \end{enumerate}
  \item \textbf{Addressing}: Multiaddr format (e.g., \texttt{/ip4/1.2.3.4/tcp/4001/p2p/12D3Koo...})
  \item \textbf{Application protocol}: \texttt{/gever/1.0.0} --- each message sent via a new stream (not persistent connection)
  \item \textbf{DHT}: Supported but \textbf{disabled} (\texttt{enableDht = false}). SGN never used Kademlia for peer discovery.
  \item \textbf{mDNS}: \textbf{Enabled} (\texttt{enableMdns = true}). Provided automatic local network peer discovery without BLE.
\end{itemize}

\subsubsection{LibP2P NAT Traversal Stack}

LibP2P provided a multi-layered NAT traversal strategy:

\begin{enumerate}
  \item \textbf{AutoNAT v2}: Probes reachability by asking other peers to dial back. Classifies the local NAT type (public, restricted, symmetric).
  \item \textbf{Circuit Relay v2}: Public peers volunteer as relays. NATed peers reserve relay slots. Traffic flows: A $\to$ Relay Peer $\to$ B. \textit{Unlike dedicated relay servers, these are libp2p peers themselves.}
  \item \textbf{AutoRelay}: Automatically discovers and connects to available relay peers when AutoNAT detects the node is behind NAT.
  \item \textbf{DCUtR (Direct Connection Upgrade through Relay)}: After establishing a relayed connection, peers coordinate hole-punching to upgrade to a direct connection.
\end{enumerate}

\textbf{Address priority hierarchy} (highest to lowest):
\begin{enumerate}
  \item IPv6/UDX --- best performance, globally routable
  \item IPv4/UDX --- common, good performance
  \item IPv4/TCP --- IPFS interop, wider firewall compatibility
  \item Circuit Relay --- last resort, highest latency
\end{enumerate}

\subsubsection{LibP2P Bootstrap Requirement}

LibP2P \textbf{required bootstrap peers} for relay discovery:

\begin{verbatim}
/ip4/40.160.9.115/tcp/4001/p2p/12D3KooWKnDdG3...
/ip6/2604:2dc0:101:100::138f/tcp/4001/p2p/12D3KooWKnDdG3...
\end{verbatim}

These bootstrap peers served as:
\begin{itemize}
  \item Initial connection points for newly started nodes
  \item Relay reservation targets (via Circuit Relay v2)
  \item AutoNAT probing partners
\end{itemize}

Without the bootstrap peer being reachable, a NATed node could not discover relays and was limited to direct connections or mDNS.

\subsubsection{LibP2P Peer Discovery}

LibP2P did \textbf{not} discover peers autonomously (DHT was disabled). Discovery happened via:
\begin{enumerate}
  \item \textbf{BLE ANNOUNCE packets} --- peers discovered via BLE include their libp2p multiaddrs
  \item \textbf{mDNS} --- automatic discovery on the same LAN (no BLE required)
  \item \textbf{Out-of-band exchange} --- manual address sharing (QR code, link)
  \item After discovery, the app called \texttt{connectToHost(hostId, hostAddrs)} with learned addresses
\end{enumerate}

\subsection{T1 Analysis (LibP2P)}

\textbf{Key insight}: When peers are co-located, \textbf{BLE is always available} regardless of network configuration. BLE provides the reliable connectivity floor.

LibP2P adds value beyond BLE when co-located:
\begin{itemize}
  \item Higher throughput than BLE (UDX/TCP vs GATT characteristic writes)
  \item \textbf{mDNS} discovers peers on the same LAN without needing BLE at all---useful when devices are on the same WiFi but beyond BLE range
  \item TCP fallback works through firewalls that block UDP
\end{itemize}

\textbf{Cross-protocol weakness}: When one peer has only IPv4 and the other only IPv6, libp2p requires a dual-stack relay peer. Since relay peers are other libp2p nodes (not dedicated infrastructure), dual-stack availability is not guaranteed.

\textbf{Critical dependency}: Without BLE, the \textit{only} transport is libp2p. If either peer lacks a routable IP address (BLE-only or link-local IPv6), communication is \textbf{impossible}.

When DCUtR hole-punching fails, libp2p falls back to Circuit Relay. Relay always works as long as both peers can reach the relay peer. However:
\begin{itemize}
  \item \textbf{Relay availability}: Depends on the bootstrap peer being online. If bootstrap is down, NATed peers have no relay.
  \item \textbf{Symmetric NAT $\leftrightarrow$ Symmetric NAT}: Relay-only; no direct upgrade possible. Higher latency ($\sim$50--150ms added round-trip depending on relay location).
  \item \textbf{Single relay}: In practice, SGN used primarily one bootstrap peer as relay---a single point of failure.
\end{itemize}

\subsection{T3 Analysis (LibP2P)}

LibP2P's Circuit Relay v2 allowed \textit{any public peer} to act as a \textbf{transport-level} relay:

\begin{itemize}
  \item A third peer C with a public IP could transparently relay encrypted libp2p connections between A and B
  \item C never sees application-level messages---all traffic is Noise-encrypted end-to-end
  \item In practice, SGN primarily relied on the bootstrap peer as the relay, not arbitrary third peers
  \item This meant the relay capacity could \textit{theoretically} grow organically with the network (more public peers = more potential relays)
\end{itemize}

\textbf{Limitation}: LibP2P's transport relay only worked when C had a \textit{public IP} and both A and B could establish libp2p connections to C. It did not help with cross-transport scenarios (e.g., A reaches C via BLE, C reaches B via libp2p). Application-level message forwarding was not implemented in this era.

\textbf{This is libp2p's biggest weakness for mobile}: Neither UDX nor TCP support connection migration. When the device's IP changes:
\begin{enumerate}
  \item All existing connections (UDX and TCP) are severed instantly
  \item The node must reconnect to the bootstrap peer from scratch
  \item AutoNAT must re-probe reachability
  \item If NATed, a new relay reservation must be made
  \item All peer connections must be re-established from scratch
  \item \textbf{Total recovery time: 10--30 seconds} of complete blackout
\end{enumerate}

\textbf{BLE-only implications}:
\begin{itemize}
  \item \textbf{Range}: $\sim$10--30m indoors, $\sim$100m outdoors per hop. With peer relay, effective range doubles to $\sim$60m indoors / $\sim$200m outdoors (single hop through a middle peer).
  \item \textbf{Throughput}: BLE GATT writes are $\sim$1--2 KB/s effective. Fine for text messages, insufficient for media. Relay adds latency but not throughput reduction (bottleneck is the BLE link itself).
  \item \textbf{Resilient}: Works in disaster scenarios, underground, in Faraday cages, on planes---anywhere Bluetooth radio works. Peer relay extends this resilience beyond direct BLE range.
\end{itemize}

\section{LibP2P Gap Analysis}
% =============================================================================

\begin{enumerate}
  \item \textbf{No connection migration}: The single biggest problem. Mobile users on cellular experience constant disconnections. Train/car scenarios are effectively unusable.

  \item \textbf{Bootstrap SPOF}: One server down = entire relay infrastructure gone for NATed peers. Adding bootstrap peers is possible but requires configuration changes.

  \item \textbf{Cross-protocol bridging}: IPv4$\leftrightarrow$IPv6 requires a dual-stack relay peer. Since relays are other libp2p peers, dual-stack availability is not guaranteed.

  \item \textbf{Double identity mapping}: LibP2P PeerId $\neq$ SGN public key creates complexity and potential for identity confusion.

  \item \textbf{Relay peer heterogeneity}: Circuit Relay depends on other peers volunteering relay capacity. In a small network, relay availability is unpredictable.

  \item \textbf{No application-level forwarding}: LibP2P's Circuit Relay only worked at the transport level and required the relay peer to have a public IP. No cross-transport forwarding (e.g., BLE$\to$libp2p bridge) was possible.

  \item \textbf{Discovery requires BLE or mDNS}: Remote peers that have never been co-located and are not on the same LAN cannot discover each other without out-of-band exchange.
\end{enumerate}

% =============================================================================
\section{Migration to Iroh}
% =============================================================================

SGN migrated from libp2p to \textbf{iroh} to address the gaps identified above. This section describes the iroh transport and evaluates how each connectivity scenario changed.

\subsection{Iroh Transport Mechanics}

\begin{itemize}
  \item \textbf{Protocol}: QUIC over UDP. Encrypted, multiplexed, and supports \textbf{connection migration}.
  \item \textbf{Identity}: NodeId = Ed25519 public key. The same key used for BLE identity. No double mapping.
  \item \textbf{Addressing}: A peer is addressed by \texttt{(NodeId, relay\_url, direct\_addresses[])}.
  \item \textbf{Relay servers}: Operated by iroh.computer (e.g., \texttt{https://use1-1.relay.iroh.network.}). They:
    \begin{itemize}
      \item Relay encrypted QUIC packets between peers who cannot reach each other directly
      \item Coordinate hole-punching attempts for NAT traversal
      \item Are \textbf{dedicated infrastructure}, not other peers
      \item Do NOT inspect, store, or modify message content
    \end{itemize}
  \item \textbf{No bootstrap required}: Iroh does not need bootstrap peers. Peer discovery is handled by BLE ANNOUNCE or out-of-band.
  \item \textbf{No mDNS}: Iroh does not provide local network discovery. BLE or out-of-band exchange is required.
  \item \textbf{Connection lifecycle}:
    \begin{enumerate}
      \item Peer A knows Peer B's NodeId + relay URL (from BLE ANNOUNCE or out-of-band)
      \item A connects to B via the relay server
      \item Iroh attempts hole-punching in the background
      \item If hole-punch succeeds, connection migrates to direct UDP
      \item If hole-punch fails, relay continues to carry traffic
    \end{enumerate}
  \item \textbf{QUIC connection migration}: When a device changes IP (e.g., WiFi $\to$ cellular), QUIC migrates the connection to the new address without re-establishing. This is the key advantage over libp2p.
\end{itemize}

\textbf{Key improvement}: Under iroh, SGN now supports \textbf{application-level peer forwarding}. This is fundamentally different from libp2p's Circuit Relay:

\begin{itemize}
  \item \textbf{Cross-transport}: C can receive via BLE from A and forward via iroh to B (or vice versa). LibP2P's Circuit Relay only worked within the libp2p transport.
  \item \textbf{Any peer can relay}: C does not need a public IP. A NATed peer connected to both A and B via any transport can forward.
  \item \textbf{End-to-end encrypted}: C sees only the encrypted envelope. Cannot read message content.
  \item \textbf{Single hop}: Only A$\to$C$\to$B is allowed. No multi-hop chains (A$\to$C$\to$D$\to$B).
  \item \textbf{Real-time only}: If C cannot reach B right now, the forward fails immediately. No queuing.
\end{itemize}

\textbf{Why QUIC migration matters}: QUIC connections are identified by a connection ID, not by the 4-tuple (src IP, src port, dst IP, dst port). When the device's IP changes:
\begin{itemize}

\end{itemize}

For a mobile-first application where users are walking, commuting, and switching networks constantly, this difference is transformative.

The two transports (BLE + Internet) are \textbf{complementary}: BLE provides resilient local connectivity that is immune to network configuration, while the Internet transport provides global reach that is immune to physical proximity. Together they cover most real-world scenarios.

The migration from libp2p to iroh was a net positive for mobile-first use cases. The loss of mDNS and TCP fallback is real but narrow; the gain in connection migration (QUIC), architectural simplicity, and relay reliability is broad and impactful. The addition of \textbf{application-level peer forwarding} under iroh recovers and exceeds libp2p's transport-level relay capability: any peer can now forward end-to-end encrypted messages across any transport combination (BLE$\to$iroh, BLE$\to$BLE, iroh$\to$iroh), limited to a single relay hop and real-time delivery only (no store-and-forward).

\section{Introduction}
% =============================================================================

SGN uses two transport layers---\textbf{BLE} (short-range, no Internet) and \textbf{libp2p} (UDX/TCP over UDP, global reach)---plus \textbf{application-level peer relay} where a connected peer C forwards end-to-end encrypted messages between A and B when no direct path exists.

\textbf{Key constraints}: (1)~\textbf{Fair message delivery}: every \texttt{send()} is eventually delivered---the sender queues locally when the recipient is temporarily unreachable; (2)~Single-hop relay only (A$\to$C$\to$B, no multi-hop); relay peers do NOT queue---forwarding is real-time only; (3)~Transport priority: BLE $\to$ libp2p (direct) $\to$ peer relay.

\textbf{Address types}: BLE-only (no IP), Private IPv4 (PIv4, behind NAT), Public IPv4 (PuIv4), Link-local IPv6 (LLv6, not routable), Public IPv6 (PuIv6).

\textbf{Topologies}: T1~(co-located, $<$30m), T2~(remote), T3~(relay peer ``friend in the middle''), T4~(GLP server / supernode).

% =============================================================================
\section{Transport Mechanics}
% =============================================================================

\textbf{BLE}: 10--100m range, no IP required. Each device advertises a GATT service UUID with a fixed 8-byte Grassroots prefix (enables scan filtering) and an 8-byte agent suffix derived from the Ed25519 public key (SHA-256 hash). After connection, ANNOUNCE packets are exchanged (full 32-byte public key, nickname, IP addresses).

\textbf{LibP2P}: UDX (primary, UDP-based) and TCP (fallback) transports, secured with \textbf{Noise XX} (Curve25519 DH, ChaCha20-Poly1305, SHA-256) providing mutual authentication and forward secrecy. LibP2P generates a \texttt{PeerId} from the Ed25519 key; the SGN public key remains the canonical identity. DHT disabled; mDNS enabled for LAN discovery. NAT traversal via GLP-server-coordinated UDP hole-punching (see Section~\ref{sec:glp-server}).

\textbf{Application-level peer relay}: Any connected peer C can forward messages across transport boundaries (e.g., BLE$\to$libp2p). C sees only encrypted envelopes. Single hop, real-time only (C does not queue). The \emph{sender} queues locally if no path exists, preserving fair message delivery.

\subsection{T4: GLP Server}
\label{sec:glp-server}

A \textbf{GLP server} is an ordinary SGN agent that happens to have a \textbf{globally routable IP address}. It requires no special protocol---it is a regular peer in the social graph that speaks the same networking API. Its role is \emph{signaling}, not relaying application data.

\textbf{Who can be a GLP server?} Any friend with a globally routable address:
\begin{itemize}[nosep]
  \item \textbf{Smartphones on carriers with public IPv6} (increasingly common; many LTE/5G carriers assign PuIv6)
  \item \textbf{Devices on networks with public IPv4} (less common for mobile, common for fiber/business)
  \item \textbf{Dedicated always-on servers} (for reliability, but not required by the architecture)
\end{itemize}

\textbf{GLP server responsibilities} (signaling only):
\begin{enumerate}[nosep]
  \item \textbf{Address registration}: NATed agents connect and report their current public IP:port.
  \item \textbf{Address notification}: When A wants to reach B, the GLP server provides A with B's registered address (and vice versa).
  \item \textbf{Hole-punch coordination}: Instructs both agents to begin simultaneous UDP sends, opening NAT mappings.
\end{enumerate}

\textbf{Smartphones as GLP servers}: A phone with PuIv6 IS directly reachable---it can receive incoming UDP packets and coordinate hole-punching between two NATed friends. It is less reliable than a dedicated server (goes offline, changes carriers), but this is mitigated by befriending \emph{multiple} GLP server peers. The architecture is federated: no single server is required, and agents with public IPs naturally emerge as signaling nodes.

\textbf{Symmetric NAT fallback}: When hole-punching fails (both peers behind symmetric NAT), the GLP server itself can forward messages via \emph{application-level peer relay}---it is just another friend in the social graph. This avoids the need for a separate Circuit Relay mechanism.

\textbf{No SPOF}: Unlike a dedicated bootstrap peer, the GLP server model is inherently redundant. An agent befriends multiple GLP servers; if one goes offline, others provide signaling. Any new friend with a public IP automatically becomes a potential GLP server.

\subsubsection{GLP Server Discovery}

How does agent A know which friends currently have a routable address and can serve as GLP servers?

\begin{enumerate}[nosep]
  \item \textbf{ANNOUNCE packets}: Each peer includes its current IP addresses in ANNOUNCE (sent over BLE and libp2p). Friends that receive the ANNOUNCE see whether the address is globally routable (PuIv6 or PuIv4) and can mark that peer as a potential GLP server.
  \item \textbf{Address change notifications}: When a peer's network changes (WiFi$\to$LTE, gaining/losing PuIv6), it pushes an updated address set to all connected friends. This lets friends dynamically track which peers are currently routable.
  \item \textbf{Probing}: If A has a stale address for friend C, A attempts a UDX connection. If it succeeds, C is reachable and can serve as GLP server. If it fails, A tries the next known GLP server friend.
\end{enumerate}

The set of available GLP servers is inherently dynamic---friends gain and lose routable addresses as they move between networks. Redundancy (befriending multiple GLP server-capable peers) absorbs this churn.

\subsubsection{Mutual Friend Requirement}

For hole-punch coordination, A and B need a \emph{mutual} GLP server friend S that is simultaneously connected to both:

\begin{itemize}[nosep]
  \item S must know both A's and B's current public IP:port (both must have registered with S).
  \item If A and B have no mutual GLP server friend, hole-punching cannot be coordinated, and communication falls back to app-level relay through whatever peers connect A to B.
  \item In practice, the social graph grows organically through BLE encounters. Two remote friends A and B likely share mutual friends (the social graph is dense locally). The introduction protocol (part of the GLP application layer, not networking) can explicitly introduce friends to GLP server peers.
\end{itemize}

\subsubsection{IPv6 Firewalling}

Some carriers assign globally routable IPv6 addresses but \textbf{firewall incoming UDP}. The phone has a PuIv6 address but cannot actually receive unsolicited packets. This creates a false positive: the peer appears routable but is not.

\textbf{Detection}: Each peer must probe its own reachability. When a peer detects it has a PuIv6 address, it asks a connected friend to attempt an incoming UDP connection. If the connection succeeds, the peer is genuinely routable and advertises itself as a GLP server candidate. If it fails (firewalled), the peer does not advertise GLP server capability.

This is analogous to libp2p's AutoNAT: a peer asks others to ``dial me back'' to verify reachability. The probe can be implemented as a simple ping/pong exchange: peer C asks friend D to send a UDP packet to C's public address; if C receives it, C is routable.

\textbf{No connection migration is libp2p's biggest weakness for mobile.} When IP changes, all UDX/TCP connections die and must reconnect from scratch (re-register with GLP servers, re-coordinate hole-punches). A co-located BLE peer can relay during the blackout. The sender's local queue preserves messages during the gap, ensuring fair delivery once connectivity is restored.

BLE, libp2p, and peer relay are \textbf{complementary}: BLE provides resilient local connectivity immune to network configuration; libp2p provides global reach immune to physical proximity; peer relay bridges the two. \textbf{Local queuing at the sender} ensures fair message delivery across transient disconnections.

\textbf{Primary gaps}:
\begin{enumerate}[nosep]
  \item \textbf{No connection migration}---libp2p UDX/TCP connections die on IP change. QUIC-based transport would solve this.
  \item \textbf{Cross-protocol bridging}---IPv4$\leftrightarrow$IPv6 needs a dual-stack GLP server friend; app relay partially mitigates.
  \item \textbf{Relay peer availability}---sparse networks may lack suitable relay peers; multiple GLP server friends mitigate.
  \item \textbf{Single relay hop limit}---deliberate simplicity trade-off; see future work.
  \item \textbf{PeerId $\neq$ SGN public key}---libp2p derives PeerId from the Ed25519 key, but maintaining the bidirectional mapping adds complexity.
\end{enumerate}

\textbf{Resolved gaps} (vs.\ previous analysis):
\begin{itemize}[nosep]
  \item \textbf{Bootstrap SPOF}: Eliminated. GLP servers are federated---any friend with a public IP. No single point of failure.
  \item \textbf{No store-and-forward}: Resolved. Sender-side local queuing provides fair message delivery. Relay peers still do not queue (real-time forwarding only).
\end{itemize}

\textbf{Recommendations}: (1)~Evaluate QUIC-based transport for connection migration; (2)~Unify PeerId with SGN public key if possible; (3)~Enable address exchange via QR code/link for remote-first friendships.

\section{Future Work: Geographic-Optimized Multi-Hop Routing}
% =============================================================================

The single-hop relay constraint (Section~3.3) exists because standard multi-hop routing (e.g., AODV) requires flooding-based route discovery---broadcasting RREQ to all peers. In a mobile BLE mesh with volatile topology, this is expensive and unreliable. \textbf{Geohash-guided AODV} can make multi-hop viable by constraining route discovery spatially.

\subsection{Geohash-Guided AODV}

\begin{enumerate}[nosep]
  \item \textbf{Geohash in ANNOUNCE}: Each peer includes a truncated geohash (5--6 characters, $\sim$5\,km$^2$ precision) in its ANNOUNCE packet. All peers maintain a \texttt{nodeId $\to$ geohash} mapping.

  \item \textbf{Directional RREQ}: When A needs a route to B and knows B's approximate geohash, A tags the RREQ with B's geohash. Intermediate peers only re-broadcast if their own geohash is geographically \emph{closer} to the destination (or within a configurable angular cone). This prunes the flood dramatically compared to standard AODV.

  \item \textbf{Geohash ring expansion}: If directional RREQ fails (no peers in the cone), reduce geohash precision (fewer characters = larger area) and retry---analogous to AODV's expanding ring search, but spatially informed.

  \item \textbf{Route maintenance}: Standard AODV RERR messages when a hop breaks. Geo-awareness helps select alternate next-hops that are still in the right geographic direction.
\end{enumerate}

\textbf{Why geohash?} Hierarchical (truncating characters widens the area---no distance math), prefix-matchable (shared prefix = spatial proximity), privacy-preserving (5-char $\approx$ 5\,km$^2$, reveals neighborhood, not location), and compact (5--6 ASCII characters per ANNOUNCE).

\subsection{Open Questions}

\begin{itemize}[nosep]
  \item \textbf{Location permission}: Mobile OS requires explicit consent; geohash must be opt-in.
  \item \textbf{Hybrid routing}: Geo-AODV only for BLE mesh (physical proximity); libp2p for long-range. Geographic routing is irrelevant for Internet transport.
  \item \textbf{Geohash spoofing}: A peer could lie about its geohash to attract or deflect traffic. Mitigations: cross-reference BLE RSSI, require signed geohash attestations from multiple neighbors.
  \item \textbf{TTL cap}: Practical limit likely 3--4 hops before latency and reliability degrade unacceptably.
\end{itemize}

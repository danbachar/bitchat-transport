\section{Introduction}
% =============================================================================

SGN uses two transport layers---\textbf{BLE} (short-range, no Internet) and \textbf{libp2p} (UDX/TCP, global reach)---plus \textbf{application-level peer relay} where a connected peer C forwards end-to-end encrypted messages between A and B when no direct path exists.

\textbf{Key constraints}: (1)~No store-and-forward---unreachable means failure; (2)~Single-hop relay only (A$\to$C$\to$B, no multi-hop); (3)~Transport priority: BLE $\to$ libp2p $\to$ peer relay.

\textbf{Address types}: BLE-only (no IP), Private IPv4 (PIv4, behind NAT), Public IPv4 (PuIv4), Link-local IPv6 (LLv6, not routable), Public IPv6 (PuIv6).

\textbf{Topologies}: T1~(co-located, $<$30m), T2~(remote), T3~(relay peer ``friend in the middle''), T4~(server/supernode).

% =============================================================================
\section{Transport Mechanics}
% =============================================================================

\textbf{BLE}: 10--100m range, no IP required. Each device advertises a unique service UUID derived from its Ed25519 public key. After GATT service discovery confirms an SGN peer, ANNOUNCE packets are exchanged (public key, nickname, signature).

\textbf{LibP2P}: UDX (primary) and TCP (fallback) transports, Noise encryption. Identity via separate Ed25519 keypair (PeerId $\neq$ SGN public key---requires double identity mapping). DHT disabled; mDNS enabled for LAN discovery. NAT traversal via AutoNAT v2, Circuit Relay v2, AutoRelay, and DCUtR hole-punching. Requires a bootstrap peer (\texttt{40.160.9.115}) for relay discovery.

\textbf{Application-level peer relay}: Any connected peer C can forward messages across transport boundaries (e.g., BLE$\to$libp2p). C sees only encrypted envelopes. Single hop, real-time only, no queuing. Distinct from Circuit Relay (which requires C to have a public IP and operates within libp2p only).

\textbf{Peer relay impact}: Significant for BLE-only peers. If a BLE-only Peer A has a co-located Peer C with libp2p, C can forward to the remote Peer B---transforming ``unreachable'' into reachable.

\textbf{NAT traversal}: Hole-punching (DCUtR) succeeds for most NAT combinations except Symmetric$\leftrightarrow$Symmetric (common on mobile carrier-grade NAT), which falls back to Circuit Relay. Single bootstrap peer = single point of failure.

\subsection{T4: Server / Supernode}

\textbf{Bootstrap peer}: Initial discovery, Circuit Relay reservation, AutoNAT probing. \textbf{SPOF}---if down, NATed peers lose relay, new peers cannot join.

\textbf{Always-on supernode}: Ideal app-level relay hub. Always reachable, connected to many peers. Bridges symmetric NAT, cross-protocol, and BLE-to-libp2p gaps. Requires running full \texttt{dart\_libp2p} node.

BLE, libp2p, and peer relay are \textbf{complementary}: BLE provides resilient local connectivity immune to network configuration; libp2p provides global reach immune to physical proximity; peer relay bridges the two.

\textbf{Primary gaps}:
\begin{enumerate}[nosep]
  \item \textbf{No connection migration}---libp2p connections die on IP change. A QUIC-based transport (e.g., iroh) would solve this.
  \item \textbf{Bootstrap SPOF}---single bootstrap peer = single point of failure for NATed peers.
  \item \textbf{Double identity mapping}---libp2p PeerId $\neq$ SGN public key adds complexity.
  \item \textbf{Cross-protocol bridging}---IPv4$\leftrightarrow$IPv6 needs dual-stack relay; app relay partially mitigates.
  \item \textbf{Relay peer availability}---sparse networks may lack suitable relay peers; supernode mitigates.
  \item \textbf{Single relay hop limit}---deliberate simplicity trade-off; see future work.
\end{enumerate}

\textbf{Recommendations}: (1)~Evaluate QUIC-based transport for connection migration; (2)~Deploy redundant bootstrap peers; (3)~Unify PeerId with SGN public key; (4)~Deploy always-on supernode as relay hub; (5)~Enable address exchange via QR code/link for remote-first friendships.

\section{Future Work: Geographic-Optimized Multi-Hop Routing}
% =============================================================================

The single-hop relay constraint (Section~3.3) exists because standard multi-hop routing (e.g., AODV) requires flooding-based route discovery---broadcasting RREQ to all peers. In a mobile BLE mesh with volatile topology, this is expensive and unreliable. \textbf{Geohash-guided AODV} can make multi-hop viable by constraining route discovery spatially.

\subsection{Geohash-Guided AODV}

\begin{enumerate}[nosep]
  \item \textbf{Geohash in ANNOUNCE}: Each peer includes a truncated geohash (5--6 characters, $\sim$5\,km$^2$ precision) in its ANNOUNCE packet. All peers maintain a \texttt{nodeId $\to$ geohash} mapping.

  \item \textbf{Directional RREQ}: When A needs a route to B and knows B's approximate geohash, A tags the RREQ with B's geohash. Intermediate peers only re-broadcast if their own geohash is geographically \emph{closer} to the destination (or within a configurable angular cone). This prunes the flood dramatically compared to standard AODV.

  \item \textbf{Geohash ring expansion}: If directional RREQ fails (no peers in the cone), reduce geohash precision (fewer characters = larger area) and retry---analogous to AODV's expanding ring search, but spatially informed.

  \item \textbf{Route maintenance}: Standard AODV RERR messages when a hop breaks. Geo-awareness helps select alternate next-hops that are still in the right geographic direction.
\end{enumerate}

\textbf{Why geohash?} Hierarchical (truncating characters widens the area---no distance math), prefix-matchable (shared prefix = spatial proximity), privacy-preserving (5-char $\approx$ 5\,km$^2$, reveals neighborhood, not location), and compact (5--6 ASCII characters per ANNOUNCE).

\subsection{Open Questions}

\begin{itemize}[nosep]
  \item \textbf{Location permission}: Mobile OS requires explicit consent; geohash must be opt-in.
  \item \textbf{Hybrid routing}: Geo-AODV only for BLE mesh (physical proximity); iroh/libp2p for long-range. Geographic routing is irrelevant for Internet transport.
  \item \textbf{Geohash spoofing}: A peer could lie about its geohash to attract or deflect traffic. Mitigations: cross-reference BLE RSSI, require signed geohash attestations from multiple neighbors.
  \item \textbf{TTL cap}: Practical limit likely 3--4 hops before latency and reliability degrade unacceptably.
\end{itemize}
